# Codebase Review — 2026-03-05

## Plugin Overview

This is a WordPress plugin called "Visitor Contact Collector" (VCC) that collects visitor contact info (name, email, phone) with GDPR compliance. The architecture is clean and well-structured.

---

## Architecture & Structure

The plugin follows a solid singleton-based OOP pattern across all classes:
- `visitor-contact-collector.php` — main entry point, bootstraps everything
- `includes/` — 7 class files, each with a single responsibility
- `assets/` — separate CSS/JS for frontend and admin
- `tests/` — PHP unit tests + JS tests with Jest

The class separation is good: `VCC_Database`, `VCC_Admin`, `VCC_Frontend`, `VCC_Shortcode`, `VCC_Export`, `VCC_GDPR`, `VCC_Config`.

---

## Security

**Strengths:**
- Nonce verification on form submission and export
- Capability checks (`manage_options`) before admin actions
- Input sanitization throughout (`sanitize_text_field`, `sanitize_email`, `wp_parse_args`)
- Prepared statements used for all DB queries
- `ABSPATH` guard in every file
- Rate limiting (3 submissions/IP/hour via transients)
- `orderby`/`order` whitelist validation in `get_contacts()`

**Issues:**
- `class-vcc-database.php:74` — the table existence check uses a raw interpolated variable in `SHOW TABLES LIKE '$table_name'` without `$wpdb->prepare()`. Should use `$wpdb->prepare("SHOW TABLES LIKE %s", $table_name)`.
- `class-vcc-database.php:85` — `DROP TABLE IF EXISTS $table_name` is unparameterized. Low risk since `$table_name` is derived from `$wpdb->prefix`, but worth noting.
- `class-vcc-database.php:232` — the `ORDER BY` clause is injected directly into the SQL string (e.g. `ORDER BY {$args['orderby']} {$args['order']}`). Although both values are whitelisted above, `wpdb::prepare()` doesn't cover identifier injection — this pattern is technically safe here but fragile if the whitelist is ever changed.
- `class-vcc-frontend.php:192` — `$_POST['vcc_nonce']` is accessed without checking if it's set first, which will produce a PHP notice if the key is absent.
- `class-vcc-database.php:360` — uses `date()` instead of `wp_date()` or `gmdate()`. WordPress recommends avoiding `date()` as it relies on server timezone.
- `class-vcc-database.php:409` — `$_SERVER['HTTP_USER_AGENT']` is stored raw (only sanitized later in `insert_contact`), but not truncated before the `text` column insert, which is fine for MySQL `text` type but could be an issue with very large payloads.

## Code Quality

- IP detection in both `VCC_Database::get_user_ip()` and `VCC_Frontend::get_client_ip()` is **duplicated**. This logic should live in one place (ideally a static helper or in the database class).
- `VCC_Config` loads `.env` into `$_ENV` and `putenv()` which is reasonable for a dev environment but could leak sensitive values in shared hosting environments. The config class exposes test DB credentials (`test_db_name`, `test_db_user`, `test_db_pass`) — these shouldn't be bundled into the plugin's config class loaded in production.
- `class-vcc-shortcode.php` registers an `init` hook inside a constructor that is itself called from an `init` hook. This means `add_shortcode` is hooked to `init` from within `init`, which will never fire. The `register_shortcode` method should call `add_shortcode` directly without re-hooking to `init`.
- `visitor-contact-collector.php:53` — `register_uninstall_hook` is called inside the constructor but the constructor is only instantiated during `plugins_loaded`. Uninstall hooks must be registered at the top level (not inside a hook callback), otherwise they don't fire on uninstall.
- `class-vcc-frontend.php:175` — an inline `<script>` block is output inside the shortcode's `ob_start()` buffer. This works but is better handled via `wp_footer` or `wp_add_inline_script` to avoid multiple inline scripts when the shortcode is used more than once on a page.
- `class-vcc-frontend.php:361` — CSS variable injection (`--vcc-primary-color`) uses `esc_html()` on a hex color value; `esc_attr()` would be more semantically correct here, though functionally identical for a hex string.
- The `update_contact` method (`class-vcc-database.php:316`) passes `null` as the format array for `$wpdb->update`, which means WordPress will infer formats — this is unreliable and should use explicit format strings.

---

## Frontend JavaScript

- `frontend.js` wraps everything in an IIFE with `$` but the `$` parameter is never passed in the IIFE call — `(function($) { ... })()` — meaning `$` inside is `undefined`. It should be `(function($) { ... })(jQuery)`.
- The `VCCForm` constructor is assigned to `window.VCCForm` directly, making it a global — this is intentional for the inline script that calls `new VCCForm(...)`, but it bypasses the module encapsulation of the IIFE.
- Real-time phone formatting and validation logic is solid. ARIA enhancements are a nice touch.

---

## Testing

- PHP unit tests use `WP_UnitTestCase` and cover database CRUD, frontend, admin, export, GDPR, and shortcode — good coverage intent.
- `test-database.php:70` — the test asserts `assertFalse($result)` for invalid data, but `insert_contact` returns a `WP_Error` object on failure, not `false`. The assertion will fail.
- JS tests use Jest with jsdom, which is appropriate.
- Mock WP functions class is a good approach for isolated PHP testing outside a full WP environment.
